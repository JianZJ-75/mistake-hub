# 登录鉴权方案

## 一、账户体系

### 1.1 核心规则

- **不存在注册接口**，所有用户最初都通过微信小程序登录时自动创建（Role=STUDENT）
- 管理员可在 Web 管理端将普通用户**升级为管理员**（Role=ADMIN），升级时系统自动生成密码
- 管理员可**重置**任何用户的密码
- 管理员可**修改自己**的密码
- 仅管理员拥有密码，普通用户 `cipher_password` 字段为 null

### 1.2 默认管理员

系统启动时由 `DataInitializer` 自动创建：

| 字段 | 值 |
|------|-----|
| code | `admin` |
| nickname | `管理员` |
| role | `admin` |
| password（原始） | `admin-1q2w3e4R` |
| cipher_password | `SHA-256(password)` 后用 `accountId` 做 AES 加密的结果 |

## 二、密码存储方案

### 2.1 存储流程

```
原始密码 → SHA-256 哈希 → digest → AES 加密(key=accountId) → cipher_password → 存入 DB
```

- 哈希工具：`Sha256Util.encrypt(password)` → 64 位 hex digest
- 加密工具：`EncryptionUtil.encryptWithRawKey(digest, accountId)`
- `cipher_password` 字段存储的是 **AES 加密后的 digest**，不是明文密码，也不是纯 digest

### 2.2 前端传输密码的统一规范

所有涉及密码传输的场景（登录、修改密码），前后端交互一律遵循：

```
前端：digest = SHA-256(password) → cipherDigest = AES(digest, nonce)
传输：{cipherDigest, nonceId}
后端：通过 nonceId 取 nonce（即拿即删）→ AES 解密 cipherDigest → 得到 digest
```

## 三、Nonce 机制

### 3.1 作用

Nonce（一次性随机密钥）用于**加密传输密码相关数据**，防止被截获重放。适用于登录、升级管理员返回密码、重置密码返回密码、修改密码等所有密码流转场景。

### 3.2 NonceController 接口

| 接口 | 说明 |
|------|------|
| `POST /v1/nonce/generate` | 通过 `EncryptionUtil.generateAesKey()` 生成 nonce 字段，存入 DB，返回完整 Nonce 对象（含 id 和 nonce） |
| `POST /v1/nonce/get` | 根据 id 查询 Nonce 并返回（**即拿即删**，一次性使用） |

### 3.3 Nonce 使用模式

**模式 A — 前端加密 → 后端解密**（用于：登录、修改密码）
1. 前端调用 `nonce/generate` → 获得 `{id, nonce}`，nonce 留在内存
2. 前端用 nonce 加密数据，连同 nonceId 发给后端
3. 后端通过 nonceId **内部查询并删除** nonce，解密数据

**模式 B — 后端加密 → 前端解密**（用于：升级管理员、重置密码）
1. 前端调用 `nonce/generate` → 获得 `{id, nonce}`，nonce 留在内存
2. 前端将 nonceId 发给后端
3. 后端通过 nonceId **内部查询并删除** nonce，用 nonce 加密数据返回
4. 前端用内存中的 nonce 解密

## 四、鉴权注解 (PreAuthorize)

### 4.1 设计

自定义注解 `@PreAuthorize`，标注在 Controller 方法上，通过 `Role.level` 判断权限：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface PreAuthorize {
    Role value();
}
```

### 4.2 权限判断规则

- ADMIN（level=2）：仅管理员可访问
- STUDENT（level=1）：管理员和学生均可访问

判断逻辑：`当前用户 role.level >= 注解要求的 role.level` → 放行，否则抛出 `ApiAuthorizeException`

### 4.3 AOP 切面 (PreAuthorizeAspect)

1. 拦截所有标注了 `@PreAuthorize` 的方法
2. 从 `ThreadStorageUtil` 获取当前用户 accountId
3. 查询用户 role → `Role.fromCode(role)` → 获取 level
4. 与注解要求的 `Role.level` 比较
5. 不满足则抛出 `ApiAuthorizeException`

### 4.4 使用示例

```java
@PreAuthorize(Role.ADMIN)    // 仅管理员
@PostMapping("/upgrade-admin")
public AccountUpgradeAdminResp upgradeAdmin(...) { ... }

@PreAuthorize(Role.STUDENT)  // 所有已登录用户
@PostMapping("/detail")
public AccountDetailResp detail() { ... }
```

## 五、微信小程序登录流程

### 5.1 后端时序

```
小程序                 后端                    微信服务器
 │                      │                        │
 │  wx.login()          │                        │
 │  获得 code           │                        │
 │                      │                        │
 │  POST /v1/account/login-wx                    │
 │  {code}              │                        │
 │ ───────────────────> │                        │
 │                      │  GET jscode2session    │
 │                      │  (appId, secret, code) │
 │                      │ ─────────────────────> │
 │                      │  {openid, session_key} │
 │                      │ <───────────────────── │
 │                      │                        │
 │                      │  按 openId 查 account  │
 │                      │  不存在 → 自动创建      │
 │                      │  (Role=STUDENT, 无密码) │
 │                      │                        │
 │                      │  创建 Session           │
 │                      │  (token + 过期1天)      │
 │                      │                        │
 │  返回 token          │                        │
 │ <─────────────────── │                        │
```

### 5.2 小程序前端逻辑

1. 进入小程序 → 检查本地缓存是否有 token
2. 有 token → 尝试请求受保护接口，若 403 则重新登录
3. 无 token → 调用 `wx.login()` 获取 code
4. 调用 `POST /v1/account/login-wx` 传入 code
5. 收到 token → `wx.setStorageSync('token', token)` 缓存
6. 后续所有请求在 header 中携带 `Authorization: Bearer {token}`

### 5.3 关键逻辑

- 小程序用户**无密码**，`cipher_password` 字段为 null
- 首次登录自动创建账户：生成随机 code（`wx_` + 12位随机串）、默认昵称、默认头像
- 登录成功后创建 Session 返回 token

## 六、Web 管理端登录流程

### 6.1 后端时序

```
前端                                 后端
 │                                    │
 │  1. POST /v1/nonce/generate        │
 │ ──────────────────────────────────> │
 │  返回 {id, nonce}                  │
 │ <────────────────────────────────── │
 │                                    │
 │  2. 本地计算:                       │
 │     digest = SHA-256(password)     │
 │     cipherDigest = AES(digest, nonce)
 │                                    │
 │  3. POST /v1/account/login-web     │
 │     {code, cipherDigest, nonceId}  │
 │ ──────────────────────────────────> │
 │                                    │
 │  4. 后端校验:                       │
 │     a. 按 nonceId 取 nonce，立即删除│
 │     b. AES 解密 cipherDigest       │
 │        → 得到用户传来的 digest      │
 │     c. 按 code 查 DB 拿到 account  │
 │     d. 校验 role = ADMIN           │
 │     e. AES 解密 cipher_password    │
 │        (key=accountId)             │
 │        → 得到 DB 中的 digest       │
 │     f. 比对两个 digest             │
 │                                    │
 │  5. 校验通过:                       │
 │     创建 Session (token + 过期1天)  │
 │     返回 token                     │
 │ <────────────────────────────────── │
```

### 6.2 Web 前端逻辑

1. 进入管理端 → 检查 localStorage 中是否有 token
2. 有 token → 尝试请求，若 403 则跳转登录页
3. 无 token → 显示登录表单（用户名 + 密码）
4. 用户提交 → 调用 `POST /v1/nonce/generate` 获取 `{id, nonce}`
5. 前端计算 `digest = SHA-256(password)`，`cipherDigest = AES(digest, nonce)`
6. 调用 `POST /v1/account/login-web` 传入 `{code, cipherDigest, nonceId}`
7. 收到 token → `localStorage.setItem('token', token)` 缓存
8. 跳转到管理首页

> 前端 SHA-256 使用 `crypto-js` 库，AES 使用 `crypto-js/aes`（需与后端 AES-256-CBC 算法对齐）

### 6.3 登录请求参数 (AccountLoginWebReq)

| 字段 | 类型 | 校验 | 说明 |
|------|------|------|------|
| code | String | @NotBlank | 用户名 |
| cipherDigest | String | @NotBlank | AES(SHA-256(password), nonce) |
| nonceId | Long | @NotNull | Nonce 记录 ID |

### 6.4 登录响应

直接返回 `String` 类型的 token。

## 七、修改用户角色流程

### 7.1 前置条件

- 操作者必须是管理员（`@PreAuthorize(Role.ADMIN)`）

### 7.2 逻辑分支

**升级为管理员（STUDENT → ADMIN）**：
- 需要 nonceId（用于加密生成的密码返回前端）
- `generatePwd()` 生成随机密码 → SHA-256 → AES(accountId) 存 DB
- 更新 role = ADMIN
- AES(原始密码, nonce) 加密返回，前端解密展示

**降级为学生（ADMIN → STUDENT）**：
- 不需要 nonceId
- 清除 `cipher_password`（置 null）
- 清除该用户所有 Session（强制下线）
- 更新 role = STUDENT

### 7.3 后端时序（升级场景）

```
管理端前端                              后端
 │                                       │
 │  1. POST /v1/nonce/generate           │
 │ ─────────────────────────────────────> │
 │  返回 {id: N1, nonce: "abc"}         │
 │ <───────────────────────────────────── │
 │                                       │
 │  2. POST /v1/account/change-role      │
 │     {accountId, role: "admin",        │
 │      nonceId: N1}                     │
 │ ─────────────────────────────────────> │
 │                                       │
 │  3. 后端处理:                          │
 │     a. 校验目标用户存在且角色不同      │
 │     b. 按 nonceId 取 nonce，立即删除   │
 │     c. generatePwd() → 原始密码       │
 │     d. SHA-256(密码) → digest         │
 │     e. AES(digest, accountId) → 存 DB │
 │     f. 更新 role = ADMIN              │
 │     g. AES(原始密码, nonce) → 加密密码 │
 │                                       │
 │  4. 返回 {cipherPassword}             │
 │ <───────────────────────────────────── │
 │                                       │
 │  5. 前端解密:                          │
 │     AES 解密(cipherPassword, "abc")   │
 │     → 得到原始密码，展示供复制         │
```

### 7.4 Web 前端逻辑

**升级**：
1. 管理员在用户管理页点击「升级为管理员」按钮
2. 弹出确认对话框
3. 确认后 → 调用 `POST /v1/nonce/generate` → 获取 `{id, nonce}`，nonce 存入组件状态
4. 调用 `POST /v1/account/change-role` 传入 `{accountId, role: "admin", nonceId}`
5. 收到 `{cipherPassword}` → 用组件状态中的 nonce 进行 AES 解密 → 得到原始密码
6. 弹出密码展示弹窗，包含：
   - 只读文本框显示明文密码
   - 「复制密码」按钮（调用 `navigator.clipboard.writeText()`）
   - 提示：「密码仅展示一次，请务必复制保存」

**降级**：
1. 管理员在用户管理页点击「降级为学生」按钮
2. 弹出确认对话框：「确定要降级该用户吗？该用户将被强制下线且密码将被清除。」
3. 确认后 → 调用 `POST /v1/account/change-role` 传入 `{accountId, role: "student"}`
4. 成功提示

### 7.5 请求参数 (AccountChangeRoleReq)

| 字段 | 类型 | 校验 | 说明 |
|------|------|------|------|
| accountId | Long | @NotNull | 目标用户 ID |
| role | String | @NotBlank | 目标角色（admin / student） |
| nonceId | Long | 可选 | 升级为管理员时必填 |

### 7.6 响应参数 (AccountChangeRoleResp)

| 字段 | 类型 | 说明 |
|------|------|------|
| cipherPassword | String | 仅升级为管理员时返回，AES(原始密码, nonce) |

## 八、重置密码流程

### 8.1 前置条件

- 操作者必须是管理员（`@PreAuthorize(Role.ADMIN)`）
- 可重置**任何用户**的密码（包括其他管理员和自己）

### 8.2 后端时序

```
管理端前端                              后端
 │                                       │
 │  1. POST /v1/nonce/generate           │
 │ ─────────────────────────────────────> │
 │  返回 {id: N1, nonce: "xyz"}         │
 │ <───────────────────────────────────── │
 │                                       │
 │  2. POST /v1/account/reset-password   │
 │     {accountId, nonceId: N1}          │
 │ ─────────────────────────────────────> │
 │                                       │
 │  3. 后端处理:                          │
 │     a. 校验目标用户存在且为 ADMIN      │
 │     b. 按 nonceId 取 nonce，立即删除   │
 │     c. generatePwd() → 新密码         │
 │     d. SHA-256(新密码) → digest       │
 │     e. AES(digest, accountId) → 存 DB │
 │     f. 清除该用户所有 Session          │
 │     g. AES(新密码, nonce) → 加密密码   │
 │                                       │
 │  4. 返回 {cipherPassword}             │
 │ <───────────────────────────────────── │
 │                                       │
 │  5. 前端解密:                          │
 │     AES 解密(cipherPassword, "xyz")   │
 │     → 得到新密码，展示供复制           │
```

### 8.3 Web 前端逻辑

1. 管理员在用户管理页点击「重置密码」按钮
2. 弹出确认对话框：「确定要重置该用户的密码吗？重置后该用户将被强制下线。」
3. 确认后 → 调用 `POST /v1/nonce/generate` → 获取 `{id, nonce}`
4. 调用 `POST /v1/account/reset-password` 传入 `{accountId, nonceId}`
5. 收到 `{cipherPassword}` → AES 解密 → 得到新密码
6. 弹出密码展示弹窗（同升级管理员，含复制按钮和提示）

### 8.4 请求参数 (AccountResetPasswordReq)

| 字段 | 类型 | 校验 | 说明 |
|------|------|------|------|
| accountId | Long | @NotNull | 目标用户 ID |
| nonceId | Long | @NotNull | Nonce 记录 ID |

### 8.5 响应参数 (AccountResetPasswordResp)

| 字段 | 类型 | 说明 |
|------|------|------|
| cipherPassword | String | AES(新密码, nonce) 加密后的密码 |

### 8.6 关键逻辑

- 密码由 `EncryptionUtil.generatePwd()` 随机生成
- 重置后**清除目标用户所有 Session**（`sessionService.removeByAccountId(accountId)`），强制重新登录
- 密码仅此一次展示，后端不存储原始密码

## 九、修改密码流程

### 9.1 前置条件

- 操作者必须是管理员（`@PreAuthorize(Role.ADMIN)`，只有管理员有密码）

### 9.2 后端时序

```
管理端前端                                    后端
 │                                             │
 │  1. POST /v1/nonce/generate (×2)            │
 │ ───────────────────────────────────────────> │
 │  返回 nonce1 {id: N1, nonce: "aaa"}        │
 │  返回 nonce2 {id: N2, nonce: "bbb"}        │
 │ <─────────────────────────────────────────── │
 │                                             │
 │  2. 本地计算:                                │
 │     oldCipherDigest = AES(SHA-256(oldPwd), nonce1)
 │     newCipherDigest = AES(SHA-256(newPwd), nonce2)
 │                                             │
 │  3. POST /v1/account/change-password        │
 │     {oldCipherDigest, oldNonceId: N1,       │
 │      newCipherDigest, newNonceId: N2}       │
 │ ───────────────────────────────────────────> │
 │                                             │
 │  4. 后端处理:                                │
 │     a. 按 oldNonceId 取 nonce1，立即删除     │
 │     b. AES 解密 oldCipherDigest → 老 digest │
 │     c. AES 解密 DB cipher_password          │
 │        (key=accountId) → DB digest          │
 │     d. 比对老 digest == DB digest            │
 │     e. 按 newNonceId 取 nonce2，立即删除     │
 │     f. AES 解密 newCipherDigest → 新 digest │
 │     g. AES(新 digest, accountId) → 存 DB    │
 │     h. 清除该用户所有 Session                │
 │                                             │
 │  5. 返回成功                                 │
 │ <─────────────────────────────────────────── │
```

### 9.3 Web 前端逻辑

1. 管理员在个人设置页点击「修改密码」
2. 弹出表单：旧密码输入框 + 新密码输入框 + 确认新密码输入框
3. 前端校验新密码两次输入一致
4. 提交 → 调用 `POST /v1/nonce/generate` **两次**，分别获取 nonce1 和 nonce2
5. 计算 `oldCipherDigest = AES(SHA-256(oldPwd), nonce1)`
6. 计算 `newCipherDigest = AES(SHA-256(newPwd), nonce2)`
7. 调用 `POST /v1/account/change-password` 传入四个参数
8. 成功 → 提示「密码修改成功，请重新登录」→ 清除本地 token → 跳转登录页

### 9.4 请求参数 (AccountChangePasswordReq)

| 字段 | 类型 | 校验 | 说明 |
|------|------|------|------|
| oldCipherDigest | String | @NotBlank | AES(SHA-256(旧密码), nonce1) |
| oldNonceId | Long | @NotNull | 旧密码对应的 Nonce ID |
| newCipherDigest | String | @NotBlank | AES(SHA-256(新密码), nonce2) |
| newNonceId | Long | @NotNull | 新密码对应的 Nonce ID |

### 9.5 关键逻辑

- 老密码和新密码各使用**独立的 Nonce** 加密传输
- 校验老密码：解密用户传来的 → 解密 DB 存储的 → 比对
- 存储新密码：解密用户传来的新 digest → `AES(newDigest, accountId)` → 写入 DB
- **修改后清除该用户所有 Session**（`sessionService.removeByAccountId(accountId)`），强制重新登录

## 十、Session 管理

### 10.1 SessionService 核心方法

| 方法 | 说明 |
|------|------|
| `create(Long accountId)` | 生成 token（`EncryptionUtil.generateKey()`），过期时间 = 当前时间 + 1 天，存入 session 表，返回 token |
| `getBySessionKey(String token)` | 按 token 查 session，若已过期则删除并返回 null |
| `renew(Session session)` | 刷新 expireTime = 当前时间 + 1 天 |
| `removeByAccountId(Long accountId)` | 删除该用户的所有 session（用于修改密码/重置密码后强制下线） |

### 10.2 Token 规格

- 生成方式：`EncryptionUtil.generateKey()`（基于 AES KeyGenerator 生成的 512 位随机 Base64 字符串，长度 88）
- 唯一性：session 表 token 字段有 UNIQUE 约束
- 有效期：1 天（24 小时）
- 续期策略：每次鉴权通过后自动续期 1 天

## 十一、鉴权拦截器 (RequestAuthInterceptor)

### 11.1 处理流程

1. 管理端口（Swagger）请求直接放行
2. 白名单路径直接放行（登录、nonce 等）
3. 校验 `Authorization` header 存在且以 `Bearer ` 开头
4. 提取 token → `SessionService.getBySessionKey(token)` 查 session
5. session 不存在或已过期 → 返回 403
6. `AccountService.getAccount(session.accountId)` 查用户
7. 用户不存在 → 返回 403
8. 写入 ThreadLocal：accountId、accountCode
9. `SessionService.renew(session)` 刷新过期时间
10. 请求完成后 `afterCompletion()` 清除 ThreadLocal

### 11.2 白名单路径

```yaml
auth-check:
  exclude-path-list:
    - "/v1/account/login-wx"
    - "/v1/account/login-web"
    - "/v1/nonce/generate"
```

> 注：`/v1/nonce/get` **不在**白名单中，因为该接口仅在已登录场景下使用（升级管理员、重置密码）。
