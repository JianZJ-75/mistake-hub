# 复习调度算法

## 一句话概括

系统根据艾宾浩斯遗忘曲线，为每道错题维护一个"复习阶段"，每个阶段对应一个递增的间隔天数。答对往后推，答错往前拉。每天凌晨自动从所有到期错题中选出优先级最高的 N 道，生成当天的复习计划。

---

## 核心概念

### 复习阶段（ReviewStage）

每道错题有一个 `reviewStage`（0~6），对应 7 个复习节点：

```
阶段    含义          距上次复习的间隔
─────────────────────────────────────
0       新录入         0 天（当天就要复习）
1       第 1 天复习    1 天
2       第 3 天复习    2 天
3       第 7 天复习    4 天
4       第 14 天复习   7 天
5       第 29 天复习   15 天
6       第 59 天复习   30 天
```

间隔序列：**0 → 1 → 2 → 4 → 7 → 15 → 30**，近似于艾宾浩斯曲线的复习节奏。

### 掌握度（masteryLevel）

0~100 的整数，反映学生对该题的熟练程度。初始为 0，答对 +20，答错 -15，不低于 0 不超过 100。

### 下次复习时间（nextReviewTime）

每次作答后，根据新阶段的间隔天数推算：

```
nextReviewTime = 当前时间 + intervalDays(新阶段)
```

录入错题时 `nextReviewTime = now`（阶段 0，间隔 0 天），意味着当天立刻出现在复习列表中。

## 答题后的状态流转

```
                  答对
    ┌──────────────────────────────┐
    │  newStage = min(stage+1, 6)  │
    │  mastery  = min(mastery+20, 100)
    └──────────────────────────────┘

                  答错
    ┌──────────────────────────────┐
    │  newStage = max(stage-2, 0)  │
    │  mastery  = max(mastery-15, 0)
    └──────────────────────────────┘
```

**举例**：一道题当前 stage=3（间隔 4 天），mastery=60

- 答对 → stage=4, mastery=80, 下次 7 天后复习
- 答错 → stage=1, mastery=45, 下次 1 天后复习（大幅回退，尽快再练）

**边界保护**：

| 场景 | 结果 |
|------|------|
| stage=0 答错 | stage 保持 0（不会变负数） |
| stage=6 答对 | stage 保持 6（已是最高阶段） |
| mastery=0 答错 | mastery 保持 0 |
| mastery=100 答对 | mastery 保持 100 |

## 每日任务生成流程

每天凌晨 2 点（或手动触发）为每个用户执行以下流程：

```
1. 查询该用户所有到期错题
   条件：status=有效 AND next_review_time <= 今天 23:59:59

2. 幂等排除
   查出今天已有复习计划的 mistake_id 集合，从到期列表中剔除
   → 保证重复调用不会产生重复计划

3. 计算优先级（对剩余的新错题）
   priority = 逾期天数 × 3.0
            + (100 - 掌握度) × 0.5
            + (6 - 阶段) × 2.0

4. 取 Top N
   N = 用户的 dailyLimit（默认 30）- 今天已有计划数
   按 priority 降序取前 N 道

5. 写入 review_plan 表
   每条记录：(account_id, mistake_id, planned_date=今天, status=PENDING)

6. 组装返回 + 写 Redis 缓存
   缓存 key: review:daily:{accountId}:{yyyy-MM-dd}
   TTL: 到次日凌晨 00:00 的剩余秒数
```

### 优先级公式解读

```
priority = overdueDays × 3.0 + (100 - mastery) × 0.5 + (6 - stage) × 2.0
           ─────────────────   ─────────────────────   ─────────────────
           逾期越久越紧急(权重最高)  掌握度越低越需要练     阶段越低说明越不熟
```

**实际效果**：逾期 30 天、掌握度 0、阶段 0 的题优先级 = 90 + 50 + 12 = 152（最高优先）；刚好到期、掌握度 80、阶段 5 的题 = 0 + 10 + 2 = 12（优先级低）。

## 数据模型关系

```
Account (用户)
  │
  ├── dailyLimit = 30        ← 每日最大复习量
  │
  ├── Mistake (错题) ×N
  │     ├── reviewStage      ← 当前阶段 0-6
  │     ├── masteryLevel     ← 掌握度 0-100
  │     ├── lastReviewTime   ← 上次复习时间
  │     └── nextReviewTime   ← 下次复习时间（调度核心依据）
  │
  └── ReviewPlan (复习计划) ×N
        ├── mistakeId        ← 关联错题
        ├── plannedDate      ← 计划日期
        └── status           ← PENDING → COMPLETED / SKIPPED
```

## 定时任务与幂等

### 定时任务

- **cron**: `0 0 2 * * ?`（每天凌晨 2 点）
- **分布式锁**: `@DistributedLockAnno(key="review:daily:generate", leaseTime=300)`，防多实例重复执行
- **异常隔离**: 单用户失败不影响其他用户（try-catch per user）

### 幂等保障（双重）

1. **应用层**：生成前查出当天已有计划的 mistake_id 集合，排除后再创建
2. **数据库层**：`review_plan` 表有唯一索引 `(account_id, mistake_id, planned_date)`，即使并发写入也不会重复

---

## Redis 缓存策略

| 项目 | 值 |
|------|---|
| key | `review:daily:{accountId}:{yyyy-MM-dd}` |
| value | `List<ReviewTaskResp>` 的 JSON |
| TTL | 到次日凌晨 00:00 的剩余秒数（自动过期） |
| 写入时机 | `generateDailyTasks` 执行后 |
| 失效时机 | 答题后更新错题状态时主动 evict |
| 读取时机 | 迭代 4 的 `/v1/review/today` 接口优先读缓存 |

## 一道题的完整生命周期

```
录入错题
  │  reviewStage=0, mastery=0, nextReviewTime=now
  ▼
当天出现在复习列表（stage 0 间隔 0 天）
  │  学生答对 → stage=1, mastery=20, next=now+1天
  ▼
1 天后出现（stage 1 间隔 1 天）
  │  学生答对 → stage=2, mastery=40, next=now+2天
  ▼
2 天后出现（stage 2 间隔 2 天）
  │  学生答错 → stage=0, mastery=25, next=now+0天  ← 回退到当天
  ▼
当天再次出现
  │  学生答对 → stage=1, mastery=45, next=now+1天
  ▼
... 如此循环，直到 stage=6 且 mastery 接近 100
```
