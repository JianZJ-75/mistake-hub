## 2. 性能测试

### 2.1 Redis 缓存性能测试

#### 测试目的

验证 Redis 缓存对系统配置读取和每日复习任务生成的性能提升效果。

#### 2.1.1 系统配置缓存 — Hit vs Miss

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 1 | `systemConfig_cacheHitVsMiss` | 首次读取 vs 二次读取 | cache miss 耗时 > cache hit 耗时 | ✅ |

**测试逻辑**：
- 第一次调用 `getByKey("review.intervals")`：Redis 无缓存，走 DB 查询 → cache miss
- 第二次调用：Redis 有缓存，直接返回 → cache hit
- 记录两次耗时（nanoTime），验证缓存命中后读取速度显著提升

#### 2.1.2 系统配置缓存 — 更新失效

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 2 | `systemConfig_updateEvictsCache` | 修改配置后缓存失效 | 更新后读取走 DB，值为新值 | ✅ |

**测试逻辑**：
1. 预热缓存：读取 `review.mastery_correct_delta`
2. 更新配置值为 `"25"`
3. 再次读取，验证返回新值 `"25"`（证明缓存已失效，走了 DB）
4. 还原配置值

#### 2.1.3 系统配置缓存 — 100 次读取命中率

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 3 | `systemConfig_hitRate_100reads` | 100 次连续读取 | 1 次 miss + 99 次 hit，命中率 99% | ✅ |

**测试逻辑**：
- 第 1 次读取触发 cache miss（查 DB + 写缓存）
- 后续 99 次均为 cache hit（直接从 Redis 返回）
- 缓存命中率 = 99 / 100 = **99%**

#### 2.1.4 每日复习缓存 — Hit vs Miss

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 4 | `reviewDaily_cacheHitVsMiss` | 首次生成 vs 二次获取 | miss 耗时 > hit 耗时，数据一致 | ✅ |

**测试逻辑**：
- 创建测试账号和错题（nextReviewTime 设为昨天，确保到期）
- 第一次调用 `todayTasks()`：触发任务生成 + 写缓存 → cache miss
- 第二次调用：直接从 Redis 读取 → cache hit
- 验证两次返回的任务列表大小一致

#### 2.1.5 每日复习缓存 — 失效重建

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 5 | `reviewDaily_evict_shouldRegenerate` | 手动失效后重建 | 缓存失效后重新生成，数据不为 null | ✅ |

**测试逻辑**：
1. 生成缓存：调用 `todayTasks()`
2. 手动失效：`evictDailyCache(accountId)`
3. 再次获取：触发重新生成，验证返回非 null

### 2.2 缓存性能测试总结

| 缓存场景 | 命中率 | 失效机制 | 重建机制 |
|---------|:------:|---------|---------|
| SystemConfig | 99% (100 次读取) | 配置更新时主动清除 | 下次读取时自动回填 |

**缓存 Key 设计**：
- 系统配置：`config:{configKey}`
- 每日复习：`review:daily:{accountId}:{yyyy-MM-dd}`，TTL 精确到次日 00:00:00

## 3. 安全测试

### 3.1 Nonce 重放攻击防御

#### 3.1.1 Nonce 一次性消费

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 1 | `nonce_consumeTwice_shouldFail` | 同一 Nonce 消费两次 | 第二次抛 BaseException | ✅ |

**测试逻辑**：
1. 生成 Nonce
2. 第一次 `consumeById(nonceId)` → 成功，返回 Nonce 对象
3. 第二次 `consumeById(nonceId)` → 抛出 `BaseException`（Nonce 已被删除，不存在）

**安全意义**：Nonce 消费后立即从数据库删除，防止重放攻击。攻击者截获请求后无法重复使用同一 Nonce 进行加密传输。

#### 3.1.2 Nonce 格式验证

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 2 | `nonce_format_32chars` | Nonce 生成格式 | 长度为 32 字符 | ✅ |

**安全意义**：32 字符的随机 Nonce 提供 128 位的随机性，暴力猜测的计算代价极高。

### 3.2 Session 过期处理

#### 3.2.1 过期 Session 返回 Null

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 3 | `session_expired_shouldReturnNull` | 手动过期 Session | 返回 null 并清理 DB 记录 | ✅ |

**测试逻辑**：
1. 创建账号并生成 Session
2. 手动将 `expireTime` 设为 2 天前
3. 再次查询 → 返回 null，且过期 Session 已从 DB 删除

**安全意义**：过期 Session 不仅返回 null（拒绝认证），还主动清理 DB 记录，防止过期 Token 被后续利用。

#### 3.2.2 有效 Session 正常返回

#### 3.2.3 Session 续期

### 3.3 AES-256-CBC 加密安全

#### 3.3.1 加解密完整性

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 6 | `aes_roundTrip_withAccountId` | 使用 accountId 作为 rawKey 加解密 | 解密后明文一致 | ✅ |

**测试逻辑**：
1. 创建管理员账号
2. SHA-256 哈希密码：`digest = SHA256("testPassword123")`
3. AES 加密：`cipher = encryptWithRawKey(digest, accountId)`
4. AES 解密：`decrypted = decryptWithRawKey(cipher, accountId)`
5. 验证 `decrypted == digest`

#### 3.3.2 完整登录加密流程

| # | 测试方法 | 场景 | 验证点 | 结果 |
|---|---------|------|--------|:---:|
| 7 | `loginWeb_fullFlow_nonceEncryptDecrypt` | Nonce → 加密 → 消费 → 解密 | 全流程明文一致 | ✅ |

**测试逻辑**：
1. 生成 Nonce
2. **模拟前端**：`SHA256(password)` → `AES.encryptWithRawKey(digest, nonce.nonce)`
3. **模拟后端**：消费 Nonce → `AES.decryptWithRawKey(cipher, consumed.nonce)`
4. 验证解密后的 digest 与前端计算的一致

**安全意义**：
- 密码从不以明文传输，始终经过 SHA-256 哈希
- AES 密钥为一次性 Nonce，每次登录使用不同密钥
- Nonce 消费后立即销毁，密钥不可复用

### 3.4 安全测试总结

| 安全机制 | 测试覆盖 | 防护目标 |
|---------|---------|---------|
| Nonce 一次性消费 | 重放攻击模拟 | 防止请求重放 |
| Nonce 随机性 | 32 字符长度验证 | 防止暴力猜测 |
| Session 过期清理 | 过期后返回 null + 删除 | 防止过期 Token 利用 |
| Session 续期 | 活跃用户自动延期 | 平衡安全与体验 |
| AES-256-CBC | 加解密轮回验证 | 传输层数据保护 |
| 完整登录流程 | Nonce + SHA256 + AES 全链路 | 端到端加密验证 |

## 5. 测试结论

### 5.1 性能测试结论

- Redis 缓存有效提升了系统配置和每日复习任务的读取性能，缓存命中率达 **99%**。
- 缓存失效机制正确：配置更新、复习提交后均能正确清除缓存并在下次读取时重建。
- 缓存设计采用主动失效 + 惰性加载策略，兼顾数据一致性和性能。

### 5.2 安全测试结论

- Nonce 一次性消费机制有效防御重放攻击，32 字符随机 Nonce 提供足够的随机性。
- Session 过期处理机制完善：过期 Token 拒绝认证并主动清理，活跃 Session 自动续期。
- AES-256-CBC 加密传输链路完整：前端 SHA-256 哈希 → Nonce 作为一次性密钥 → AES 加密传输 → 后端消费 Nonce 解密，全流程数据保护到位。

### 5.3 兼容性测试结论

- 微信小程序在 iOS / Android / HarmonyOS 多平台表现一致，核心功能均正常运行。
- Web 管理端在主流浏览器（Chrome / Safari / Firefox / Edge）下 UI 渲染和交互均正常。
