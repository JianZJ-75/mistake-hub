##### 2.11 POST /v1/tag/update 接口规格

```
权限: @PreAuthorize(Role.ADMIN)
请求: TagUpdateReq
  - id: Long          @NotNull("标签ID不能为空")
  - name: String      @NotBlank("标签名称不能为空")
  - type: String      @NotBlank("标签类型不能为空")    // SUBJECT / CHAPTER / KNOWLEDGE
  - parentId: Long    @NotNull("父标签ID不能为空")     // 0=顶级
响应: void
业务规则:
  1. 校验 id 对应的 Tag 存在，不存在抛 oops
  2. 校验 TagType.isValid(type)
  3. parentId != 0 时校验父标签存在
  4. 同名 + 同类型 + 同父级不允许重复（排除自身）
  5. 不允许将自己设为自己的子标签（parentId != id）
```

##### 2.12 POST /v1/tag/delete 接口规格

```
权限: @PreAuthorize(Role.ADMIN)
请求: TagDeleteReq
  - id: Long          @NotNull("标签ID不能为空")
响应: void
业务规则:
  1. 校验 Tag 存在
  2. 检查 mistake_tag 表是否有引用，有则拒绝删除，oops("该标签已被错题引用，无法删除")
  3. 检查是否有子标签（parent_id = 当前 id），有则拒绝删除，oops("该标签下有子标签，请先删除子标签")
  4. removeById
```

### 迭代 3：艾宾浩斯复习调度算法

**目标**：实现核心调度算法，自动生成每日复习任务。本迭代纯后端，无前端任务。

#### 后端

| # | 任务 | 说明 |
|---|------|------|
| 3.1 | 设计并建表 `review_plan` | 见下方表结构 |
| 3.2 | 设计并建表 `review_record` | 见下方表结构 |
| 3.3 | 创建 Entity + Mapper | ReviewPlan、ReviewRecord 实体类 + Mapper |
| 3.4 | 定义 ReviewStage 枚举 | 阶段 0~6 对应间隔 [0, 1, 2, 4, 7, 15, 30] 天，标准枚举方法集 |
| 3.5 | 定义 ReviewPlanStatus 枚举 | PENDING / COMPLETED / SKIPPED |
| 3.6 | 实现 ReviewService | 见下方服务方法规格 |
| 3.7 | 实现 ReviewScheduler 定时任务 | `@Scheduled` 每日凌晨 2 点 |
| 3.8 | Redis 缓存当日任务 | key: `review:daily:{accountId}:{yyyy-MM-dd}`，TTL 到次日凌晨 |

##### 数据表设计

```sql
CREATE TABLE review_plan (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    account_id      BIGINT      NOT NULL,
    mistake_id      BIGINT      NOT NULL,
    planned_date    DATE        NOT NULL COMMENT '计划复习日期',
    status          VARCHAR(16) NOT NULL DEFAULT 'PENDING' COMMENT 'PENDING/COMPLETED/SKIPPED',
    created_time    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_account_date (account_id, planned_date),
    INDEX idx_mistake_date (mistake_id, planned_date)
) COMMENT '复习计划表';

CREATE TABLE review_record (
    id                   BIGINT AUTO_INCREMENT PRIMARY KEY,
    account_id           BIGINT   NOT NULL,
    mistake_id           BIGINT   NOT NULL,
    review_plan_id       BIGINT            COMMENT '关联的计划ID，主动复习时可为null',
    is_correct           TINYINT  NOT NULL COMMENT '1-答对 0-答错',
    review_stage_before  INT      NOT NULL,
    review_stage_after   INT      NOT NULL,
    mastery_before       INT      NOT NULL,
    mastery_after        INT      NOT NULL,
    review_time          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_account_time (account_id, review_time)
) COMMENT '复习记录表';
```

##### ReviewService 核心方法规格

```
1. List<ReviewTaskResp> generateDailyTasks(Long accountId)
   - 查询 mistake 表: account_id = accountId AND status = 1 AND next_review_time <= today
   - 计算优先级: priority = overdue_days × 3.0 + (100 - masteryLevel) × 0.5 + (6 - reviewStage) × 2.0
   - 按 priority 降序，取 top N（N = account.dailyLimit，默认 30）
   - 为每条创建 ReviewPlan（planned_date = today, status = PENDING）
   - 组装 ReviewTaskResp 返回

2. void updateMistakeAfterReview(Long mistakeId, boolean isCorrect)
   - 读取 Mistake 当前 stage 和 mastery
   - 答对: newStage = min(stage + 1, 6), newMastery = min(mastery + 20, 100)
   - 答错: newStage = max(stage - 2, 0), newMastery = max(mastery - 15, 0)
   - nextReviewTime = now + INTERVALS[newStage] 天
   - lastReviewTime = now
   - updateById

3. void cacheDaily(Long accountId, List<ReviewTaskResp> tasks)
   - Redis SET: review:daily:{accountId}:{today} → JSON(tasks)
   - TTL: 到次日 00:00:00 的秒数

4. List<ReviewTaskResp> getCachedDaily(Long accountId)
   - Redis GET，命中返回反序列化结果，miss 返回 null
```

##### ReviewScheduler 定时任务

```
@Scheduled(cron = "0 0 2 * * ?")
void generateAllDailyPlans():
  1. 查询所有「活跃用户」：至少有 1 条 status=1 且 next_review_time <= today 的 mistake
  2. 遍历每个用户调用 generateDailyTasks(accountId)
  3. 结果写入 Redis 缓存
  4. 记录日志：共为 N 个用户生成了 M 条复习任务

幂等性：先清理当天已有的 PENDING 状态 review_plan（防止重复执行），再重新生成
```

##### ReviewTaskResp 字段

```
Long mistakeId
String title
String imageUrl
Integer reviewStage
Integer masteryLevel
Integer overdueDays        // 逾期天数
Double priority            // 优先级分数
List<TagResp> tags         // 学科信息从关联的 SUBJECT 类型标签获取 name
String planStatus          // PENDING / COMPLETED / SKIPPED
Long reviewPlanId          // 关联的 plan ID
```

**交付标准**：
- 手动调用 `generateDailyTasks(accountId)` 可正确返回优先级排序的复习列表
- 定时任务可幂等执行，不产生重复 plan
- Redis 缓存命中时不查数据库
- 无逾期错题的用户不产生空 plan
- updateMistakeAfterReview 的 stage/mastery/nextReviewTime 计算正确（单元测试覆盖边界：stage=0 答错、stage=6 答对、mastery=0 答错、mastery=100 答对）

##### 4.1 POST /v1/review/today

```
权限: @PreAuthorize(Role.STUDENT)
请求: 无（从 token 获取 accountId）
响应: List<ReviewTaskResp>
业务逻辑:
  1. 先查 Redis 缓存 getCachedDaily(accountId)
  2. 缓存 miss → 调用 generateDailyTasks(accountId) → 写入缓存 → 返回
  3. 缓存 hit → 直接返回（含已完成/跳过状态）
```

##### 4.2 POST /v1/review/submit

```
权限: @PreAuthorize(Role.STUDENT)
请求: ReviewSubmitReq
  - mistakeId: Long      @NotNull
  - isCorrect: Boolean   @NotNull
响应: ReviewSubmitResp
  - reviewStageBefore: Integer
  - reviewStageAfter: Integer
  - masteryBefore: Integer
  - masteryAfter: Integer
  - nextReviewTime: LocalDateTime
业务逻辑:
  1. 校验 mistake 存在且属于当前用户
  2. 记录当前 stage / mastery（before 快照）
  3. 调用 updateMistakeAfterReview(mistakeId, isCorrect)
  4. 创建 ReviewRecord（含 before / after 快照）
  5. 更新对应 ReviewPlan.status = COMPLETED
  6. 更新 Redis 缓存中该任务的状态
  7. 返回 before / after 对比数据
幂等性: 同一 mistakeId 当天重复提交，检查 plan 已 COMPLETED 则 oops("该题今日已完成复习")
```

##### 4.3 POST /v1/review/skip

```
权限: @PreAuthorize(Role.STUDENT)
请求: ReviewSkipReq
  - mistakeId: Long      @NotNull
响应: void
业务逻辑:
  1. 更新 ReviewPlan.status = SKIPPED（不影响 mistake 的 stage / mastery）
  2. 更新 Redis 缓存
```

##### 4.4 POST /v1/review/progress

```
权限: @PreAuthorize(Role.STUDENT)
请求: 无
响应: ReviewProgressResp
  - totalToday: Integer           // 今日总任务数
  - completedToday: Integer       // 今日已完成
  - skippedToday: Integer         // 今日已跳过
  - streakDays: Integer           // 连续复习天数
业务逻辑:
  - 从 Redis 缓存或 review_plan 表统计今日数据
  - 连续天数计算：从昨天往前追溯，每天至少完成 1 条 COMPLETED 的 plan 才算连续
```

##### 接口规格

```
5.1 POST /v1/stats/overview
权限: STUDENT
响应: StatsOverviewResp
  - totalMistakes: Integer          // 总错题数（status=1）
  - masteredCount: Integer          // 已掌握（masteryLevel >= 80）
  - pendingReviewCount: Integer     // 待复习（nextReviewTime <= today）
  - newThisWeek: Integer            // 本周新增

5.2 POST /v1/stats/subject
权限: STUDENT
响应: List<SubjectStatsResp>
  - subject: String
  - count: Integer
  - avgMastery: Double
（按 count 降序，通过 Tag(type=SUBJECT) + mistake_tag 联表聚合学科信息。未关联 SUBJECT 标签的归入"未分类"）

5.3 POST /v1/stats/mastery
权限: STUDENT
响应: MasteryDistributionResp
  - notMastered: Integer            // masteryLevel < 60
  - learning: Integer               // 60 <= masteryLevel < 80
  - mastered: Integer               // masteryLevel >= 80

5.4 POST /v1/stats/daily-completion
权限: STUDENT
请求: 无（默认近 30 天）
响应: List<DailyCompletionResp>
  - date: String (yyyy-MM-dd)
  - totalPlanned: Integer
  - completed: Integer
  - completionRate: Double          // completed / totalPlanned，无计划的天返回 null

5.5 POST /v1/stats/streak
权限: STUDENT
响应: StreakResp
  - currentStreak: Integer          // 当前连续天数
  - longestStreak: Integer          // 历史最长连续天数
（从 review_plan 表按 planned_date 聚合，至少 1 条 COMPLETED 算当日有效）

5.6 POST /v1/stats/mastery-trend
权限: STUDENT
请求: 无（默认近 30 天）
响应: List<MasteryTrendResp>
  - date: String (yyyy-MM-dd)
  - avgMastery: Double              // 当日所有复习题目的 mastery_after 平均值
（从 review_record 表按 review_time 日期聚合）

5.7 POST /v1/stats/admin-overview
权限: ADMIN
响应: AdminOverviewResp
  - totalUsers: Integer             // 总用户数
  - activeUsersToday: Integer       // 今日有复习记录的用户数
  - totalMistakes: Integer          // 全平台错题总数
  - avgCompletionRate: Double       // 全平台近 7 天平均复习完成率
```

## 六、艾宾浩斯复习调度算法

### 6.1 复习阶段与间隔

| 阶段 (stage) | 间隔 (天) | 累计天数 | 含义 |
|:---:|:---:|:---:|------|
| 0 | 0 | 0 | 新错题，当天需复习 |
| 1 | 1 | 1 | 第 1 天后复习 |
| 2 | 2 | 3 | 第 3 天后复习 |
| 3 | 4 | 7 | 第 7 天后复习 |
| 4 | 7 | 14 | 第 14 天后复习 |
| 5 | 15 | 29 | 第 29 天后复习 |
| 6 | 30 | 59 | 第 59 天后复习（已掌握） |

### 6.2 状态更新规则

```
答对：
    stage = min(stage + 1, 6)
    mastery = min(mastery + 20, 100)
    next_review_time = now + INTERVALS[new_stage]

答错：
    stage = max(stage - 2, 0)
    mastery = max(mastery - 15, 0)
    next_review_time = now + INTERVALS[new_stage]
```

### 6.3 每日任务生成优先级公式

```
priority = overdue_days × 3.0 + (100 - mastery_level) × 0.5 + (6 - review_stage) × 2.0
```

- `overdue_days`：逾期天数 = max(0, 今天 - next_review_time 的天数)
- 值越高，越优先被选入当日复习列表
- 从候选池中取 priority 最高的 N 条（N = 用户的 daily_limit）

## 七、风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|:---:|:---:|------|
| 微信小程序审核周期长 | 中 | 中 | 开发阶段使用体验版/开发版测试，提前准备审核材料 |
| 复习算法参数需调优 | 高 | 低 | 间隔参数和权重做成可配置项（迭代 6 system_config 表） |
| 前后端联调耗时 | 中 | 中 | 后端先出 Swagger 文档，前端 Mock 并行开发 |
| Redis 缓存一致性 | 低 | 中 | 写操作完成后主动失效缓存，下次读时回填 |
| 前端加密与后端不一致 | 中 | 高 | AES-CBC 参数严格对齐（IV 分隔符 `_`、PKCS5Padding） |
| 定时任务重复执行 | 低 | 中 | 先清理当天 PENDING 的 plan 再重新生成（幂等设计） |
